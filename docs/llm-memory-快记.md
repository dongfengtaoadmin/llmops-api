# LLM 记忆功能：详细说明与快记总结

> 依据「缓冲 / 窗口 / 令牌 / 摘要 / 摘要缓冲混合 / 向量库」六类常见模式整理。

---

## 共通前提：记忆是如何“实现”的

大模型单次调用通常是**无持久状态**的：应用层维护「聊天历史」，每次请求把历史与当前提问一起**填充进 Prompt**，模型才能“记得”上文。典型闭环为：

1. **记忆模块**保存历史（Human / AI 交替或等价结构）。
2. 用户发起**新提问**时，将记忆**填充**给主 Chat Model。
3. 主模型**生成** AiMessage。
4. 将本轮问答**写回**记忆模块；若超出策略限制，则**删除或压缩**旧内容。

下文各模式差异主要在：**存什么、删什么、何时删、如何压缩**。

---

## 2.1 缓冲记忆（Buffer Memory）

### 定义

最基础的记忆模型：**Human 与 AI 产生的消息全部保留**。每次新请求把**完整历史**与用户当前输入一并交给模型，从而在上下文窗口允许范围内形成**无损**对话理解。

### 流程要点

- 记忆模块中按时间堆叠 `Human: …` / `AI: …`。
- 向主模型**填充整段聊天历史**后再生成回复。
- 新生成的 AiMessage **存回**记忆，供下一轮使用。

### 优点

1. **无损记忆**：用户输入内容原则上全部保留（在未触顶上下文前）。
2. **实现简单**：兼容性好，各类大模型均支持。

### 缺点

1. **延迟与成本**：历史全部进 Prompt，token 持续上升，响应变慢、费用变高。
2. **令牌上限**：总长触及模型最大 token 后，**更早的对话无法再进入上下文**，等效“记不全”。
3. **容量有限**：上下文窗口小的模型，可用记忆长度很短。

---

## 2.2 缓冲窗口记忆（Buffer Window Memory）

### 定义

在缓冲思路上加**窗口值 k**：只保留**最近 k 次** Human/AI 对话信息；新对话写入后若超过 k，则**删除最旧的交互**，使窗口大小近似恒定。

（图示中常举例 **k=2**，即只存最近两轮对话。）

### 流程要点

- 记忆 → **填充聊天历史** → 主 Chat Model → 生成 AiMessage。
- 写回记忆时：**检查已存对话条数/轮数**，超过则删旧留新。

### 优点

1. **控 token**：历史不会无限增长，对 token 使用表现好。
2. **小模型友好**：若任务不需要久远上下文，往往效果与成本平衡较好。
3. **实现简单、性能好、各 LLM 均支持**。

### 缺点

1. **不适合需要长期记忆的交互**：较早内容会被**必然丢弃**。
2. **单条消息过长**：即使轮数在 k 以内，若单条 Human/AI **特别长**，仍可能**单独撑爆**模型上下文窗口。

---

## 2.3 令牌缓冲记忆（Token Buffer Memory）

### 定义

仍基于「缓冲」思想，但限制条件不是「轮数 k」，而是**总 token 上限 `max_tokens`**。在 Human/AI 消息序列上维护一段**随对话滚动的缓冲**：当**累计聊天历史超过 `max_tokens`** 时，开始**遗忘最早**的互动，使进入 Prompt 的文本总量受控。

### 流程要点

- 配置 **`max_tokens`**：超限则遗忘更早互动。
- 填充历史 → 主模型生成 → 写回记忆，并**同时检查**已存内容是否超 token 预算，超则删旧。

### 优点

1. **与模型上下文对齐**：可按所用模型的实际上下文长度分配记忆预算。
2. **对小模型友好**：不涉及复杂远期引用时，往往仍能保持较好效果。
3. **实现相对简单、性能好、各 LLM 支持**。

### 缺点

1. **不适合长周期、超多轮依赖**：一旦早先内容被挤出缓冲，即**硬遗忘**，无法恢复。

---

## 2.4 摘要总结记忆（Summary Memory）

### 定义

不把**原始逐条消息**长期完整塞进 Prompt，而是维护一份**记忆摘要**。后续多轮交互主要传递**摘要信息**，而非完整历史，从而在长对话中**抑制 token 膨胀**。

### 流程要点

- 记忆模块核心是 **「记忆摘要」**。
- 主对话：摘要 **填充聊天历史** + 用户提问 → 主 Chat Model → 生成 AiMessage。
- **更新摘要**：将本轮用户问题、AI 回复，连同**历史摘要**一并送入**中间 Chat Model**，由其中间模型**生成新摘要并写回**记忆模块。

### 优点

1. **长短结合**：可覆盖较长跨度互动（摘要侧偏“模糊长记忆”）。
2. **长对话更省 token**：能支撑更多轮次对话。
3. **可扩展性**：初期可能因摘要调用略费 token，但摘要增长通常**远慢于**原始历史堆叠，对话越久相对「纯缓冲」越划算。

### 缺点

1. **细节损失**：具体措辞与细粒度事实可能在概括中丢失。
2. **短对话不划算**：轮数很少时，摘要链路可能**比直接发原文更费 token**。
3. **依赖中间模型**：摘要质量完全取决于**中间 LLM** 能力，且摘要调用本身占用 token。

---

## 2.5 摘要缓冲混合记忆（Summary Buffer Hybrid Memory）

### 定义

**摘要记忆**与**缓冲窗口记忆**的结合：既保留**对话摘要**（长期、压缩），又保留**最近若干轮原文**（短期、精确）。与「按轮数截断的简单窗口」不同，常见做法是：以 **Token 长度**为阈值——当近期缓冲超过限制时，将**较早的原文**交给中间模型**并入摘要**，并从缓冲中清掉已摘要部分。

### 流程要点

- **记忆模块**同时包含：**近期原文缓冲**（Human/AI 块）+ **记忆摘要**。
- 主模型侧：通常同时注入 **历史摘要 + 近期原文**，再回答用户。
- 当缓冲超过 **Token 限制**：将待归档的旧消息与**已有摘要**送入**中间 Chat Model**，**生成新摘要并更新**；缓冲区释放空间。
- 另有一条摘要更新路径：新问答也可参与摘要迭代（与 2.4 类似的「中间模型更新摘要」逻辑）。

### 优点

1. **混合精度**：长期靠摘要（略模糊），近期靠原文（精确）。
2. **长对话 token 效率**：比全量缓冲更能「记住」更多轮次的语义信息。

### 缺点

1. **远期仍是摘要**：时间久远的内容在摘要中**细节不保真**。
2. **额外成本**：摘要更新需要**额外 LLM 调用**与工程复杂度。

---

## 2.6 向量存储库记忆（Vector Store Memory）

### 定义

将记忆以向量形式存入**向量数据库**。每次请求用当前问题（或查询文本）做**相似度检索**，取 **Top-K** 条最相关的历史片段，**填充进 Prompt**，再交给主模型。本质是**语义检索 + 选择性注入**，而非全历史或纯时间窗口。

### 流程要点

- 用户提问同时触发：对向量库 **检索 k 条**相关历史。
- 检索结果与用户问题一起 **填充 Prompt** → Chat Model 生成。
- 新生成的内容**写回向量库**（需先 **Embedding** 等），供后续检索。

### 特点（与摘要等对比）

- 可承载**极大规模**历史（近似“容量上可扩展很大”）。
- 相比纯摘要，往往**更易保留与查询相关的具体片段**（取决于切块与检索质量）。
- **即时连贯性**通常弱于「整段缓冲」全量塞入（因每次只取 Top-K 相关条）。

### 优点

1. **细节与容量**：较摘要更利于保留**相关细节**；较简单缓冲更能扩展**可存储对话规模**。
2. **Token 相对均衡**：只带**相关片段**进模型，而非整段历史。

### 缺点

1. **性能与复杂度**：需要 **Embedding、向量库维护、检索链路**，整体更重。
2. **检索质量决定记忆质量**：检索好则效果很好，检索差则**偏差大**，“两极分化”。

---

## 模式对照（便于选型）

| 模式 | 控制维度 | 长期信息 | 实现复杂度 | 典型代价 |
|------|----------|----------|------------|----------|
| 缓冲 | 无 / 仅受上下文顶满 | 全保留至上限 | 低 | token 线性涨 |
| 缓冲窗口 | 最近 k 轮 | 丢远期 | 低 | 低 |
| 令牌缓冲 | `max_tokens` | 丢远期 | 低～中 | 低～中 |
| 摘要 | 摘要文本 | 压缩长程 | 中 | 中间模型调用 |
| 摘要+缓冲混合 | Token + 摘要 | 摘要+近期原文 | 中高 | 中间模型 + 缓冲 |
| 向量库 | Top-K 语义 | 依赖检索命中 | 高 | Embedding + 库运维 |

---

## 快记总结

- **本质**：LLM 默认不“记住”会话；**记忆 = 应用层维护状态 + 每次请求塞进 Prompt**。
- **2.1 缓冲**：全量原文，**最简、最真**，但 token 与上下文上限是瓶颈。
- **2.2 窗口（k 轮）**：只留最近 k 轮，**控长度**，**远期必丢**；注意**单条超长**仍可撑爆上下文。
- **2.3 令牌缓冲**：按 **`max_tokens`** 滑动删最旧，**与模型窗口对齐**，仍是**挤出即永久丢失**。
- **2.4 摘要**：用**中间模型**滚动更新摘要，**长跑省 token**，但**细节有损**，短对话可能不划算。
- **2.5 摘要+缓冲混合**：**摘要管长期 + 原文管近期**，常以 **Token 阈值**触发把旧缓冲并入摘要；**细节与成本**折中最好，但工程与多一次模型调用。
- **2.6 向量库**：**Embedding + Top-K 检索**注入上下文，**容量大、相关细节好**，但依赖**检索质量**，系统更复杂。

**一句话选型**：要极简用缓冲；要稳态成本用窗口或令牌缓冲；要长跑主题用摘要或混合；要海量相关回忆用向量库。

---

*具体参数（k、`max_tokens`、摘要频率、切块大小、K 值、嵌入模型）需结合业务与所用模型的上下文窗口调优。*
