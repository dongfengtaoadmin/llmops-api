# 图片转 Prompt 与 HTML 生成方案

## 1. 背景与问题

使用大语言模型将 UI 截图直接转换为 HTML，通常存在以下问题：

- 对于复杂 UI，即使经过多轮迭代优化，也不一定能达到理想效果，尤其是包含复杂交互的非内容型网站。
- LLM 对颜色和组件位置的识别成功率相对较低，容易输出错误颜色，或者错误定位组件。
- 以 GPT-4o 为例，截图中的实践结论认为英文 Prompt 的效果通常优于中文 Prompt。

从另一个角度看，LLM 生成结果较差通常可以归结为两个因素：

1. 模型本身的视觉理解或代码生成能力不足。
2. 输入给模型的 Prompt 不够完整、准确和细致。

当无法更换更强的模型时，可以重点优化输入：不要直接要求模型“照着截图写 HTML”，而是先把截图转换成尽可能详细、结构化的 UI 描述，再根据描述生成代码。

## 2. 核心思路：先图片转 Prompt，再 Prompt 转 HTML

图片转 HTML 可以拆成两个阶段：

```text
UI 截图 → 结构化 UI Prompt → HTML/CSS/JavaScript
```

这样，问题就从“模型能否一次看懂图片并写出完整代码”，转化为“如何尽可能准确、详细地描述图片”。

详细 Prompt 应覆盖以下信息：

- 页面整体结构与区域划分。
- 组件类型，例如 `button`、`input`、`div`、`img`、`video`。
- 组件的位置、尺寸、间距和对齐方式。
- 父子关系、层级关系与排列顺序。
- 颜色、背景色、字号、字体、边框、圆角、阴影等样式。
- 文本、图片等组件内容。
- 点击、输入、切换、弹窗等交互行为。
- 响应式布局和不同状态下的表现。

## 3. 图片转 Prompt 的处理流程

### 3.1 OCR 与组件识别

首先使用本地 OCR 或视觉模型识别截图中的组件，得到初步组件列表及其元数据，例如：

```json
{
  "type": "button",
  "content": "提交",
  "position": { "x": 120, "y": 360 },
  "size": { "width": 96, "height": 40 },
  "parent": "form-panel",
  "styles": {
    "color": "#ffffff",
    "backgroundColor": "#1677ff",
    "fontSize": "14px",
    "borderRadius": "6px"
  },
  "behavior": "点击后提交表单"
}
```

### 3.2 拆分组件并多次分析

遍历 OCR 得到的组件列表，把组件逐个拆分，并交给视觉模型进行多轮识别。不同轮次可以分别关注：

- UI 设计规范：结构、位置、尺寸、颜色、字体、边框等。
- 行为模式：点击、输入、切换、悬停、弹窗和页面跳转等。
- 组件关系：父子组件、相邻组件、分组和层叠关系。

这样可以为每个组件得到更完整的描述信息。

### 3.3 合并完整 Prompt

把以下内容合并成一个完整 Prompt：

- 英文预设 Prompt。
- 页面整体描述。
- 每个组件的详细描述。
- 设计规范与交互行为。
- 技术栈、输出格式和约束条件。

对于复杂 UI，最终 Prompt 可能达到数万甚至十万 Token。因此需要选用支持长上下文输入的模型，并允许代码分多轮输出。

> 截图中列举了 GPT-4o、GPT-4o mini、o1、Qwen Max 和 Qwen Plus 的上下文及输出 Token 数据。这些属于截图制作时的记录，模型规格可能发生变化，实际使用前应查询对应平台的最新官方文档。

## 4. HTML 续写机制

### 4.1 为什么需要续写

模型允许的输入上下文通常大于单次最大输出长度。完整 UI Prompt 可以一次性作为输入，但复杂页面的 HTML、CSS 和 JavaScript 未必能在一轮内完整输出。

因此可以将生成过程设计成循环：

```text
生成一部分 → 判断是否完成 → 未完成则携带状态续写 → 再次判断
```

### 4.2 推荐流程

1. 将完整 Prompt 传给 LLM，开始生成页面。
2. 保存本轮生成的代码和生成进度。
3. 判断输出是否包含约定的完成标记。
4. 如果已经完成，进行 HTML 校验并输出最终结果。
5. 如果尚未完成，将已有代码、剩余任务和中断位置传给下一轮。
6. 重复执行，直到完成或达到最大续写次数。

### 4.3 续写 Prompt 示例

```text
Continue implementing the HTML page from the exact point where the previous
generation stopped.

Requirements:
1. Do not repeat code that has already been generated.
2. Preserve the existing structure, naming, styles, and technology choices.
3. Complete the unfinished components and interactions listed below.
4. Ensure all opened HTML tags, CSS blocks, and JavaScript blocks are closed.
5. If the page is fully complete, append this exact marker at the end:
   <!-- GENERATION_COMPLETE -->

Original UI specification:
{full_ui_prompt}

Existing generated code:
{previous_html}

Last stable generation point:
{checkpoint}

Remaining tasks:
{remaining_tasks}
```

### 4.4 建议使用结构化返回值

为了让程序稳定判断是否需要续写，可以要求模型返回结构化状态：

```json
{
  "completed": false,
  "content": "本轮新增的代码",
  "checkpoint": "正在实现右侧设置面板",
  "remaining_tasks": [
    "补全右侧设置面板",
    "实现确认弹窗",
    "增加移动端响应式样式"
  ]
}
```

程序仅把 `content` 追加到结果中，并根据 `completed` 决定是否继续。

### 4.5 续写实现注意事项

- 不要只发送“请继续”，否则模型容易重复或偏离原设计。
- 每轮携带完整设计要求、已有结果、检查点和剩余任务。
- 设置最大续写次数，避免模型无法完成时无限循环。
- 每轮保存检查点，调用失败后可以从最近一次成功结果恢复。
- 拼接前检查代码是否重复，必要时按稳定边界截断。
- 完成后校验 HTML 标签、CSS 语法和 JavaScript 语法。
- 最后应实际渲染页面，再与原截图进行视觉比对和迭代修正。

## 5. 多 Agent 协同方案

这个流程适合使用多个 Agent 分工完成。每个 Agent 负责一个边界清晰的任务，通过结构化结果传递信息。

### 5.1 Agent 角色划分

| Agent | 职责 | 主要输出 |
| --- | --- | --- |
| OCR Agent | 识别文字和初始组件 | 组件列表、文字、坐标和尺寸 |
| Layout Agent | 分析页面结构与组件关系 | 页面区域、父子关系、布局约束 |
| Style Agent | 识别视觉设计规范 | 颜色、字体、间距、边框、阴影 |
| Interaction Agent | 推断组件交互行为 | 点击、输入、切换、弹窗等行为 |
| Prompt Composer Agent | 汇总所有分析结果 | 完整、结构化的英文 UI Prompt |
| Code Generator Agent | 根据 Prompt 生成页面 | HTML、CSS 和 JavaScript |
| Continuation Agent | 管理中断和续写 | 检查点、剩余任务、新增代码 |
| Reviewer Agent | 校验完整性和视觉一致性 | 问题列表、修改建议、完成状态 |

### 5.2 Agent 协作流程

```text
UI 截图
  ↓
OCR Agent：识别组件和基础元数据
  ↓
Layout / Style / Interaction Agents：并行补充详细描述
  ↓
Prompt Composer Agent：合并英文预设 Prompt 和组件描述
  ↓
Code Generator Agent：生成首轮 HTML
  ↓
Reviewer Agent：判断是否完整、是否符合截图
  ├─ 完整 → 输出最终结果
  └─ 未完整 → Continuation Agent 整理检查点和剩余任务
                 ↓
             Code Generator Agent 继续生成
                 ↓
             再次交给 Reviewer Agent
```

### 5.3 协作原则

- 每个 Agent 的输入和输出尽量采用固定 JSON Schema。
- 分析型 Agent 可以并行执行，代码生成与审查应串行循环。
- Prompt Composer 负责消除不同 Agent 描述之间的冲突。
- Reviewer 不直接盲目重写全部代码，只输出具体问题和待办事项。
- Continuation Agent 负责找到稳定续写边界，避免重复代码。
- 全流程保留中间结果，方便追踪错误来自 OCR、描述、生成还是续写阶段。

## 7. 总结

图片转 HTML 的重点不是让一个模型在一次调用中完成所有事情，而是构建一条可拆分、可检查、可恢复的生成流水线：

1. 使用 OCR 和视觉模型识别组件。
2. 通过多个 Agent 分别分析布局、样式和交互。
3. 合并成详细、结构化的英文 Prompt。
4. 使用代码生成 Agent 分阶段生成页面。
5. 通过完成标记、检查点和剩余任务实现可靠续写。
6. 由审查 Agent 校验完整性，并通过渲染对比持续优化。

核心原则可以概括为：**先提高图片描述的准确度，再提高代码生成的完整度；以结构化协作代替一次性生成。**
